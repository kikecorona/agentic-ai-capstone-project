Project Overview

Objectives

What We Set Out to Build

01

Single Source of Truth

A continuously-updated knowledge base that gives engineers, product managers, and leadership an accurate, current map of system architecture, dependencies, and ownership — without manual maintenance.

02

Collapse Gap-Analysis Lead Time

Reduce dependency discovery from weeks of manual archaeology to on-demand answers. Any team should be able to ask "what services implement this product?" and get a grounded, cited answer in seconds.

03

Autonomous Refresh, Human-Grounded

The system runs continuously — triggered by code commits or a schedule — enriching pages from source code and existing docs. When it can't resolve a gap, it escalates to an SME rather than hallucinating.

04

Apply Agentic AI Patterns at Scale

Demonstrate a production-grade multi-agent architecture using: supervisor/specialist pattern, MCP contracts for isolation, LangGraph for stateful loops, Tree of Thoughts for non-trivial decisions, and RAG for grounded retrieval.

Target users:   Engineering leads · Product managers · New-hire onboarding · Cross-team dependency reviews

Design Principles

Guiding Principles for the Agent

🔒

Read-Only on External Systems

The agent only reads source code, telemetry, Quip, Confluence, and GitHub issues. It writes exclusively to the BP/SD documentation pages it owns and to its own embedding store. No side-effects on production systems.

📋

Git-Backed Pages

All generated docs live in a versioned Git repo. Every page write is a commit. Regressions roll back with git revert and re-indexing falls out of the next scheduled refresh automatically.

👥

Human-in-the-Loop

When the agent can't resolve a gap with confidence, it escalates to a subject-matter expert through the Portal — never during a user query, only during background builds. SME answers feed back into the index, closing the loop.

⛓

Bounded Loops — No Runaway Agents

Every cycle has an explicit cap: Auto-RAG rewrites bounded to R=2, ToT beam search to B=2-3 / D=2-3. After the cap the loop returns a degraded-but-safe outcome (low_confidence or exhausted), never recurses further.

🔌

MCP Contracts for Isolation

Agents never call peers directly — all traffic flows through typed MCP contracts. Each agent is wired to a fixed tool set. Adding a new specialist is a registration + one routing rule; existing agents are unaffected.

🔭

Observable by Default

Every inbound/outbound MCP call emits an OpenTelemetry span. Every LLM call is logged to an audit DB. Payload summaries are captured (counts, IDs, status) but never raw query text or document content.

Problem & Four-Agent Architecture

Pain Points in a 20+ Year Org
  • Stale documentation — written once, never updated after the first sprint
  • Knowledge silos — Quip, Confluence, wikis, tribal memory, email threads
  • Late dependency discovery — integration failures found at release, not design
  • No architecture map — nobody can answer "what does this service own?" confidently
Why Four Agents?
  • B&P + SD split — different reasoning, different source material, different page shapes
  • Orchestrator — centralised routing, SME queue, state; specialists stay focused
  • RAG Service — shared embedding model, one consistent retrieval behaviour; new specialists get it free

  • Corrective — SME escalation closes gaps with human-verified content; SME answers re-index automatically

A synthetic Apple App Store–style e-commerce system built deliberately with uneven implementation and documentation gaps — realistic input data for the agent's gap-detection workflows.

Auto-RAG Loop — Self-Correcting Retrieval

4-Node LangGraph StateGraph
  • Decide — classify into no_retrieval | retrieve; honour domain_filter
  • Retrieve — top-K=10 similarity search; domain filter pushed to vector query
  • Grade — LLM scores each doc 0-3; survivors go to answer generation
  • Rewrite — reformulates query; may widen domain_filter (bp → both)
Loop Bounds & Outcomes
  • Bounded to R=2 rewrites
  • ok — faithfulness + answerability both pass
  • low_confidence — best-effort answer, rewrites exhausted
  • exhausted — no usable evidence at all
Index-Quality Feedback Loop

Chunks that repeatedly fail grading emit an index-quality flag in the retrieval trail. The calling specialist re-indexes that source on the next refresh — closing the loop between retrieval and indexing.

Two-Axis Re-Grade

Faithfulness — every claim supported by a cited source (hallucination detection).
Answerability — the answer substantively addresses the query.

Why ToT for Chunking?

Chunking strategy is a high-leverage, low-frequency decision: a bad split hurts every future retrieval from that document. Alternatives are cheap to generate and scoring is reliable (probe Q&A similarity), making this an ideal ToT application point.

ToT is not used for every decision — only where intermediate steps are non-trivial, alternatives are cheap to draft, and scoring is reliable.

Candidate Strategies
per-paragraph per-section per-N-chars summary-only hybrid

Domain prose differences (BP narrative vs SD structured) flow through naturally via content-driven probe questions.

Specialist Visibility

Specialists only see chunking_strategy + embedding_revision in the response. The entire ToT sub-graph is encapsulated inside the RAG Service and reached only via RAG_MCP.index.

SD runs to completion before BP so BP sees a fresh SD doc-index when it runs.

Two Parallel Tracks
  • Enrich loop — iterates existing SD pages, fills gaps using fresh source analysis
  • New service track — detects services in source code with no SD page; runs full compose-from-scratch using analyze_code + ToT dep graph
Skip-Unchanged Optimisation

Per-page skip iff page_hash + side_info_revision both match. side_info_revision hashes the source analysis, so a code change re-enriches even when page bytes didn't move. force=true bypasses both checks.

Parallel Fan-Out

Orchestrator hands one job per page to specialists in parallel. Refresh time ∝ slowest page, not total page count. Per-page ordering preserved so cross-references stay consistent within a refresh cycle.

analyze_code Extracts (Python AST)
  • Endpoints — Flask routes, Blueprint mounts
  • Data structures — @dataclass, type hints, Pydantic models
  • Data stores — SQLAlchemy tables, raw CREATE TABLE strings
  • Downstream calls — requests HTTP calls, sqlite3 queries; dynamic targets tagged for SME review

  • telemetry-reweighted — volume-weighted edges from Monitoring MCP
  • stable-history — prefers fewer dynamic edges

Query vs Background Mode

Query mode: always returns an answer (low-conf flagged if uncertain), never escalates to SME.

Background mode: escalates unresolvable gaps to SME queue via placeholder blocks.

Cross-Domain Isolation

B&P calls SD MCP only for relational queries ("what services serve this product"). SD calls BP MCP for the reverse. Neither writes into the other's pages.

Similarity retrieval always goes through RAG MCP — never a direct peer call.

Cross-Reference Validation

All relative Markdown links to SD pages are re-validated on every refresh. Broken links become follow-up tasks. Answer-time resolution ensures users see up-to-date links even if the persisted page is briefly stale.

Design Intent

Tight system prompt (~300 tokens) and a small, deterministic action space keep the orchestrator reliable on the local LLM. Deep domain reasoning stays inside specialists — the orchestrator only picks which specialist to call and assembles results.

Action Space
  • dispatch_to_bp
  • dispatch_to_sd
  • dispatch_to_both — parallel fan-out
  • ingest_sme_reply
  • ack_completion — updates SME queue
  • done
Resumable State

Every node persists state before transitioning. Mid-refresh crash picks up from the last completed node. Only pending_sme_questions survives between runs — doc indexes and sources inventories are owned by the specialists.

Why Merge at the Orchestrator?

Specialists answer independently from their domain. The orchestrator is the only node with both responses — it dedupes sources, collapses near-identical prose, and runs a final editorial pass before the answer reaches the user.

Status Precedence
ok
›
low_confidence
›
exhausted

The response inherits the strongest status — if one specialist returned ok, the merged answer is ok.

Dedup + Collapse Logic

Sources deduped by (source_uri, domain). Near-identical answers (substring containment or ≥85% token overlap) collapsed to avoid repeating the same prose twice. A From BP + SD: lead-in is preserved in the merged body.

PoC Scope

Proof-of-Concept Limitations

Model & Quality
  • Single local model — all nodes share llama3.1:8b; no per-node specialization
  • No adversarial testing — prompt injection and PII leakage rates unmeasured
Infrastructure
  • Single-host — no horizontal scaling or load balancing
  • SQLite everywhere — not production-grade at volume
  • In-process ChromaDB — no dedicated vector store
  • Stub OTel collector — SQLite-backed, not a real backend
Triggers & Sources
  • No webhook — schedule-based or manual only
  • Git only — Slack, Confluence, email out of scope
  • Source code not in RAG — only BP docs and SD pages indexed
Synthetic Corpus
  • Hand-crafted test data — real legacy codebases will surface edge cases
  • Partial OTel coverage — only 5 of 10 services emit spans
Security & Access
  • No auth — anyone who can reach the Portal can trigger refreshes
  • Single shared repo — no per-domain access control
  • Stub SME registry — no skill-based routing or workload balancing
Evaluation
  • No eval harness — hallucination checks are manual spot-checks
  • Monitoring MCP absent — ToT falls back to code references for dep-graph scoring
