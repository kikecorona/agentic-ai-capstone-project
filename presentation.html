Problem

Problem Definition

Pain Points in a 20+ Year Org
  • Stale documentation — written once during the first sprint, never updated as the system evolves; decisions get made on outdated assumptions
  • Knowledge silos — tribal knowledge spread across Quip, Confluence, wikis, email threads, and people who have since left
  • Late dependency discovery — integration failures and missing dependencies surface at release time, not during design
  • No architecture map — nobody can confidently answer "what services implement this product?" or "who calls this endpoint?"
  • Cross-org black boxes — upstream and downstream dependencies are poorly documented, making it hard to prioritize or measure integration risk
Why a Standalone LLM Falls Short
  • Multiple data sources — source code, telemetry, Quip, Confluence, wikis, APIs; no single LLM context can hold all of it
  • Incomplete & inaccurate input — documents are stale, conflicting, or missing entirely; raw retrieval amplifies the problem
  • No verification layer — a single model can't distinguish a correct claim from a plausible-sounding stale one without grounding
  • No human escape hatch — when confidence is low, there's no mechanism to escalate to an expert and close the gap
The Opportunity

Modernisation efforts in a 20+ year old org stall on one recurring problem: nobody has an accurate, current map of the system. A Research Agent that continuously updates and enriches documentation collapses gap-analysis lead time from weeks of manual archaeology to on-demand answers — giving every team a single source of truth they can trust.

Project Overview

Objectives

What We Set Out to Build

01

Single Source of Truth

A continuously-updated knowledge base that gives engineers, product managers, and leadership an accurate, current map of system architecture, dependencies, and ownership — without manual maintenance.

02

Collapse Gap-Analysis Lead Time

Reduce dependency discovery from weeks of manual archaeology to on-demand answers. Any team should be able to ask "what services implement this product?" and get a grounded, cited answer in seconds.

03

Autonomous Refresh, Human-Grounded

The system runs continuously — triggered by code commits or a schedule — enriching pages from source code and existing docs. When it can't resolve a gap, it escalates to an SME rather than hallucinating.

04

Apply Agentic AI Patterns at Scale

Demonstrate a production-grade multi-agent architecture using: supervisor/specialist pattern, MCP contracts for isolation, LangGraph for stateful loops, Tree of Thoughts for non-trivial decisions, and RAG for grounded retrieval.

Target users:   Engineering leads · Product managers · New-hire onboarding · Cross-team dependency reviews

Design Principles

Guiding Principles for the Agents

🔒

Read-Only

The agent's purpose is to gather, organise, and surface knowledge — not to act on it. It reads source code, telemetry, and documentation, and writes only to the documentation pages it owns. No side-effects on production systems, ever.

🔄

Always Up to Date

The knowledge base refreshes continuously as code and docs change. Every run re-checks freshness, re-grades existing pages, and closes stale gaps automatically — documentation that drifts gets corrected on the next cycle without manual intervention.

👥

Human-in-the-Loop

When confidence is low the agent surfaces the gap to a subject-matter expert through the Portal rather than inventing an answer. SME replies enrich the index directly — closing the loop and preventing the same question from being escalated twice.

🛡️

Secure & Privacy-Aware

No PII data is retained at any stage. Access is scoped to the organisation's own systems. The LLM runs locally (llama3.1:8b on Ollama) so no document content leaves the network. Call logs capture counts and status — never raw text.

🔌

Extensible by Design

Every agent communicates through typed MCP contracts. Adding a new specialist — a new data source, a new domain — is a registration and a routing rule; existing agents are untouched. The architecture grows without rewiring the core.

🔭

Observable by Default

Every MCP call emits an OpenTelemetry span. Every LLM call is audited. Grader-fail rate, escalation rate, and index-quality hit rate are tracked in the dashboard — making invisible drift visible before it becomes a problem.

What Is Pear Store?

A synthetic Apple App Store–style e-commerce system built deliberately with uneven implementation and documentation gaps — realistic input data for the agent's gap-detection workflows.

  • 10 microservices — 8 storefront + 2 PearCare warranty

Architecture

Architecture Overview

Orchestrator Supervisor

Routes work and holds all cross-agent state — no content of its own. It routes Portal queries to the right specialist, forwards refresh events from the Update Trigger, and deduplicates SME questions by topic.

  • Owns — pending SME questions queue, specialist registry, routing rules
  • Does not own — any documentation content, indexes, or embedding pipeline; all deep work is delegated to specialists
  • Interaction pattern — receives task envelopes from Portal or Trigger, returns structured responses or escalations from specialists
B&P + SD Specialists Domain Experts

Two separate agents, each owning a distinct documentation domain with different source material and reasoning requirements. Cross-domain references flow through typed MCP calls — never direct function calls.

  • B&P owns — BP pages in GitHub, BP doc index (content hash, open placeholders, embedding revision), BP sources inventory
  • SD owns — SD pages in GitHub, SD doc index, SD sources inventory; reads source code via GitHub MCP and telemetry via Monitoring MCP
  • Neither embeds directly — both delegate chunking, embedding and retrieval to the RAG Service via RAG_MCP.index() and RAG_MCP.retrieve()
RAG Service Shared Infrastructure

The fourth agent is not a content specialist — it's shared retrieval infrastructure that both specialists depend on. Co-locating embedding, chunking, and the Auto-RAG loop here means consistent retrieval behaviour and a free upgrade path for any future specialist.

Owns

Shared Embeddings DB · embedding model · Auto-RAG loop · ToT chunking-strategy sub-graph

Does not own

Source code access · GitHub writes · SME state · any per-page metadata; stateless across requests except the vector store

Domain isolation

All chunks carry a domain tag (bp | sd). Cross-domain queries drop the filter — no merge step, no peer-MCP call

    FB --> Out

4-Node LangGraph StateGraph
  • Decide — no_retrieval vs. retrieve; applies domain_filter
  • Retrieve — top-K=10 similarity; domain filter on vector query
  • Grade — LLM scores docs 0–3; survivors feed answer generation
  • Rewrite — reformulates query; widens domain_filter if needed
Loop Bounds & Outcomes
  • Bounded to R=2 rewrites
  • ok — faithfulness + answerability both pass
  • low_confidence — best-effort, rewrites exhausted
  • exhausted — no usable evidence found
Index-Quality Feedback Loop

Repeated grade failures emit an index-quality flag. Caller re-indexes that source on the next refresh.

Two-Axis Re-Grade

Faithfulness — claims backed by cited sources.
Answerability — answer addresses the query.

Why ToT for Chunking?

Chunking strategy is a high-leverage, low-frequency decision: a bad split hurts every future retrieval from that document. Alternatives are cheap to generate and scoring is reliable (probe Q&A similarity), making this an ideal ToT application point.

ToT is not used for every decision — only where intermediate steps are non-trivial, alternatives are cheap to draft, and scoring is reliable.

Candidate Strategies
per-paragraph per-section per-N-chars summary-only hybrid

Domain prose differences (BP narrative vs SD structured) flow through naturally via content-driven probe questions.

Specialist Visibility

Specialists only see chunking_strategy + embedding_revision in the response. The entire ToT sub-graph is encapsulated inside the RAG Service and reached only via RAG_MCP.index.

SD runs to completion before BP so BP sees a fresh SD doc-index when it runs.

Two Parallel Tracks
  • Enrich loop — iterates existing SD pages, fills gaps using fresh source analysis
  • New service track — detects services in source code with no SD page; runs full compose-from-scratch using analyze_code + ToT dep graph
Skip-Unchanged Optimisation

Per-page skip iff page_hash + side_info_revision both match. side_info_revision hashes the source analysis, so a code change re-enriches even when page bytes didn't move. force=true bypasses both checks.

Parallel Fan-Out

Orchestrator hands one job per page to specialists in parallel. Refresh time ∝ slowest page, not total page count. Per-page ordering preserved so cross-references stay consistent within a refresh cycle.

analyze_code Extracts (Python AST)
  • Endpoints — Flask routes, Blueprint mounts
  • Data structures — @dataclass, type hints, Pydantic models
  • Data stores — SQLAlchemy tables, raw CREATE TABLE strings
  • Downstream calls — requests HTTP calls, sqlite3 queries; dynamic targets tagged for SME review

  • code-only — static call analysis
  • telemetry-reweighted — volume-weighted edges from Monitoring MCP
  • stable-history — prefers fewer dynamic edges

Query vs Background Mode

Query mode: always returns an answer (low-conf flagged if uncertain), never escalates to SME.

Background mode: escalates unresolvable gaps to SME queue via placeholder blocks.

  • write_bp_doc
  • rag_retrieve
  • compose_answer
  • escalate — background mode only
  • done
Cross-Domain Isolation

B&P calls SD MCP only for relational queries ("what services serve this product"). SD calls BP MCP for the reverse. Neither writes into the other's pages.

Similarity retrieval always goes through RAG MCP — never a direct peer call.

Cross-Reference Validation

All relative Markdown links to SD pages are re-validated on every refresh. Broken links become follow-up tasks. Answer-time resolution ensures users see up-to-date links even if the persisted page is briefly stale.

Design Intent

Tight system prompt (~300 tokens) and a small, deterministic action space keep the orchestrator reliable on the local LLM. Deep domain reasoning stays inside specialists — the orchestrator only picks which specialist to call and assembles results.

Action Space
  • dispatch_to_bp
  • dispatch_to_sd
  • dispatch_to_both — parallel fan-out
  • ingest_sme_reply
  • ack_completion — updates SME queue
  • done
Resumable State

Every node persists state before transitioning. Mid-refresh crash picks up from the last completed node. Only pending_sme_questions survives between runs — doc indexes and sources inventories are owned by the specialists.

Why Merge at the Orchestrator?

Specialists answer independently from their domain. The orchestrator is the only node with both responses — it dedupes sources, collapses near-identical prose, and runs a final editorial pass before the answer reaches the user.

The response inherits the strongest status — if one specialist returned ok, the merged answer is ok.

Dedup + Collapse Logic

Sources deduped by (source_uri, domain). Near-identical answers (substring containment or ≥85% token overlap) collapsed to avoid repeating the same prose twice. A From BP + SD: lead-in is preserved in the merged body.

  • Hand-crafted test data — real legacy codebases will surface edge cases

Security & Access
  • No auth — anyone who can reach the Portal can trigger refreshes

Evaluation
  • No golden-set benchmark — RAG grades per-answer faithfulness, but no regression suite tracks quality over time

Wrap-Up

Takeaways & Learnings

Key Takeaways
Good system design + small local LLMs can deliver production-quality results — llama3.1:8b achieved 88.4% re-grade pass rate & 82.5% ToT success with 0 errors across 1,500 calls
Using LLMs as processing primitives — judges, classifiers, Q&A generators — unlocks structured knowledge from raw unstructured docs at scale
LLM latency is a first-class design constraint in multi-agent systems — loop bounds and beam width aren't just safety rails, they're performance decisions
Software engineering best practices (typed contracts, OTel tracing, domain isolation) are directly transferable to agentic systems and make them debuggable
Observability was the key feedback loop — index-quality hit rate, grader-fail rate & escalation counts surfaced gaps no static analysis would have caught
AI-assisted coding made this scale possible — a project of this scope wouldn't be achievable in the available timeframe without it; used with intent and understanding, it remains a genuinely didactic experience
What I Learned
Multi-agent system design — agent isolation, loop bounds, SME-in-the-loop for confidence gaps, domain-separated ownership, and OTel observability as a core architectural concern
RAG retrieval tuning is critical for quality — query parameters have a large impact on output; moving from TOP_K=4 to TOP_K=25 was one of the highest-leverage improvements in the pipeline
Multi-agent systems require dedicated infrastructure — embedding databases (ChromaDB), agent-oriented observability platforms (OTel + audit logs), and typed inter-agent contracts aren't optional; they're load-bearing
Agentic AI toolbox hands-on:
ReAct loops · MCP contracts · Tree-of-Thought · LangGraph · Auto-RAG · Local LLMs (Ollama)
What's still missing: a dedicated eval harness to catch quality regressions across pipeline changes — the LLM-as-grader works well at enrichment time but doesn't guard against silent degradation when prompts, models, or retrieval parameters change
